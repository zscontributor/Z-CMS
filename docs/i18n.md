# Translating Z-CMS

**English is the base language.** Every key exists in English first; every other
language falls back to it, key by key.

That fallback is the reason you can contribute a translation without finishing
it. A language that covers 60% of the catalogue shows 60% in your language and
the rest in English — never a blank screen, never a crash. So a partial
translation is worth merging, and you are not signing up for a marathon.

## The two catalogues

Z-CMS keeps platform strings and theme strings apart, and they are contributed to
in different places.

| | Where it lives | What is in it | Who ships it |
| --- | --- | --- | --- |
| **Core** | `packages/i18n/src/locales/` | The admin UI, the API's error messages, the public runtime's own pages | The platform |
| **Theme** | `themes/<name>/src/locales/` | Everything a theme says: "Read more", "Page 2 of 5" | The theme |

They are separate on purpose. A theme is installed, updated and removed
independently of the platform — if its strings lived in core, translating a theme
would require a release of Z-CMS, and deleting a theme would leave dead keys
behind forever. It is also not the same job: the person who localises "Read more"
for a magazine theme is not the person who localises "Row-Level Security is not
enabled on this table."

## Adding a language to the core

Two things, and neither of them is TypeScript: a folder of translated JSON under
`packages/i18n/src/locales/<code>/`, and one line in `packages/i18n/locales.json`.
Then `pnpm --filter @zcmsorg/i18n sync`.

**The step-by-step lives next to the files you edit:
[packages/i18n/README.md](../packages/i18n/README.md#adding-a-language-step-by-step).**
It is the only copy on purpose — a second one here would be wrong within a
release, and the version a contributor actually reads is the one in the folder
they are already standing in.

The rest of this page is what that README does not cover: the two catalogues, how
to translate a **theme**, and which strings belong in a catalogue at all.

## Flags

Every language switcher shows a flag beside the native name. A contributor adding
a language writes nothing to get one — it is derived from the locale code, and the
[README](../packages/i18n/README.md#the-flag) covers the two cases where you would
override that.

What matters everywhere else in the codebase is the rule the feature rests on:
**the flag is never the label.** A language is not a country — Arabic is spoken in
twenty of them, Esperanto in none — so `flagFor()` returns `null` for the
languages no flag can honestly represent, and every surface that renders one
renders it next to the name that does the naming, and stays correct when it is
absent. A switcher that shows only flags asks the reader "which country owns your
language", and for a great many of them the answer is "none of these".

The SVGs come from [flag-icons](https://github.com/lipis/flag-icons) (MIT) and are
copied into each app's `public/z-flags/` at build time — generated, gitignored,
never imported. A browser fetches the two or three a switcher paints; the other
268 sit on disk. Importing flag-icons' stylesheet instead would ship all of them
to every reader.

### For theme authors

The flag arrives resolved, on the alternate:

```tsx
ctx.alternates.map((a) => (
  <a href={a.path} hrefLang={a.locale}>
    {a.flagUrl && <img src={a.flagUrl} alt="" aria-hidden="true" />}
    {localeName(a.locale)}
  </a>
))
```

`flagUrl` is `string | null`, and null is ordinary — handle it, do not treat it as
an error. **Do not build a language-to-country table in a theme.** A site's
languages are rows in its database, chosen after your theme shipped; a table in
the theme would need a release every time somebody added Norwegian. That is the
same reason `themes/default` names languages with `Intl.DisplayNames` rather than
a map of its own.

## Why the catalogue is generated, and why there are two entrypoints

A browser must never download a language nobody is reading. That sounds obvious
and is easy to get wrong: `import { catalog } from "@zcmsorg/i18n"` inside a client
component compiles, works, and quietly ships **every language the project has
ever merged** to every user. Nothing breaks. The admin just gets heavier with each
translation PR — so the project is punished, in page weight, for succeeding at
i18n.

So the package has two doors:

| | Exports | For |
| --- | --- | --- |
| `@zcmsorg/i18n` | translator, locale metadata, **and the catalogue** | server: cms-api, server components |
| `@zcmsorg/i18n/client` | translator, locale metadata | anything with `"use client"` |

`@zcmsorg/i18n/client` does not re-export the catalogue, so the mistake above is not
a code-review item — it is an import that does not resolve.

What a client component gets instead is one locale, resolved on the server by
`messagesFor(locale)` with the English fallback already folded in, handed down
through `LocaleProvider`. It arrives in the RSC payload with the first HTML: no
network round trip, no flash of untranslated UI, and a payload whose size depends
on how much text the admin contains — not on how many languages exist.

That is also why `catalog.ts` is generated rather than written. It only encodes
*structure* (which locales exist, which namespaces each one actually ships), and
structure is exactly what a script can derive from the filesystem without a human
forgetting a line.

## Translating a theme

A theme carries its own messages, in its own package:

```
themes/default/src/locales/
├── en.json      # the base
├── vi.json
└── ja.json
```

Adding one is one file plus one line in the theme's `defineTheme` call:

```ts
import de from "./locales/de.json";

export default defineTheme({
  messages: { en, vi, ja, de },
  …
});
```

That is the whole change. Core is not involved, and no other file in the repo
needs to know.

Templates read these through `ctx.t`, never through the core catalogue:

```tsx
<a href={ctx.url(item.path)}>{ctx.t("archive.readMore")}</a>
```

## Which strings are translated, and which are not

Not every string in the codebase belongs in a catalogue, and putting the wrong
ones there makes the catalogue worse.

**Translated** — anything a person reads in a browser: admin UI, API error
messages (cms-api negotiates the language from the request's `Accept-Language`
header), the public 404 page, theme text.

**Not translated** — operator and developer output: log lines, `console.warn`
diagnostics, CLI output, exceptions that only ever reach a terminal. An engineer
reading a stack trace at 3am is not helped by a log line in a language the rest of
the stack trace is not in.

Content is not translated either. A post written in Vietnamese stays Vietnamese —
that is *content*, and it belongs to the site, not to the interface.

## Style

- Translate the **meaning**, not the words. If the English is clumsy in your
  language, write the sentence a native speaker would write.
- Keep the register consistent: Z-CMS speaks plainly to an administrator, without
  exclamation marks and without apologising.
- Do not translate identifiers, technical tokens, or product names that appear
  inside a message (`X-Site-Id`, `PUBLISHED`, `Z-CMS`).
- If a key's English is genuinely ambiguous, that is a bug in the English. Open an
  issue rather than guessing — you will not be the last person to hit it.
